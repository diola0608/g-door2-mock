# 01. ドメインモデル（背景知識）── Composer EDU

> [00-principles.md](00-principles.md) の「深く推測する」を支えるのが、この背景知識です。
> ここを内在化していれば、ユーザーが言わなかった画面・状態・導線を自分で補完できます。
> 用語は画面ラベルにそのまま使うので、表記を揃えてください（末尾の用語集参照）。

---

## 1. システムの目的と規模

教育委員会向けの大規模システム。3つのサービス群からなる：

1. **名簿サービス** … 名簿レジストリ、所属変更申請、CSVインポート、処理状況・ログ
2. **組織・アカウント** … 組織管理、アカウント管理（Workspace）、権限ポリシー
3. **フォームサービス／データ利活用** … フォーム作成・配信・集計、分析ダッシュボード

規模感：1テナント（県）で 20万人超（生徒18万＋教職員2.8万規模）、組織600件超、属性項目40件超。
→ 一覧は必ずページネーション・検索・フィルタ前提。件数バッジ（`214,387人` 等）を出す。

---

## 2. テナンシー（マルチテナント）

- **1県 = 1テナント = 1つの Google GCP プロジェクト** としてデプロイ。
- **データベースはテナントごとに別 GCP プロジェクト** → 他テナントのデータと混ざらない（物理分離）。
- 画面では：ヘッダーのテナント切替（`長野県教育委員会` + 緑ドット）、サイドバー脚注
  （`テナント: nagano-pref-edu / GCP: composer-nagano-prod`）で「分離されている」ことを示す。

---

## 3. 組織階層（ツリー）

テナント（県）をルートとした階層構造。**枝の形は一定でない**（学校種が市町村の下にも県直下にもある）：

```
県（テナント全体）
├─ 市町村 ┬─ 学校種 ─ 学校
│         └─ 教育委員会 ─ 部署
├─ 学校種 ─ 学校
└─ 学校（県直轄の県立高校 等）
```

- アイコン語彙：県=`flag`／市町村=`location_city`／学校種=`category`／学校=`school`／
  教委=`corporate_fare`／部署=`meeting_room`。
- 各ノードに所属人数カウントを併記（`.tree .cnt`）。
- 「組織スコープ」はこのツリーで選ぶ／表す（`roster.html` 左ペインが参照実装）。

---

## 4. 管理対象（人）

1. **教育委員会に所属する人々**（教委職員）
2. **教員**
3. **生徒とその関係者**（保護者など）

- 人はメールアカウントを持つ場合と持たない場合がある。
- 種別の色：生徒=`chip indigo`／教員=`chip green`／教委職員=`chip sky`／関係者=`chip gold`。
- 人の表示は必ず `.person`（アバター＋氏名＋かな/ID）で統一。

---

## 5. Google Workspace 連携（一方向同期）

- 管理対象のワークスペースに属するメールを持つ人は、Workspace アカウントとデータ同期する。
- **同期は このシステム → Google Workspace の一方通行。** システムに登録された項目が
  Workspace 側アカウントへ反映される（逆方向はない）。
- 反映は **定期バッチ（例：毎日 04:00）** が基本。即時同期はオプション。
- 画面では：`.sync`（同期済 `cloud_done`／反映待ち `sync`／対象外 `cloud_off`）、
  OUパス（`/長野市/小学校/城山小/生徒/2021入学`）、一方向であることのcallout。

## 6. 属性項目（テナント独自のカスタム項目）

名簿は固定項目に加え、**テナントが自由に項目を追加できる**（＝属性項目）。
例：生年月日、所属部活、教員免許番号、戸籍、不登校歴…。

各属性項目に設定するもの：

- **データ型（5種）**：
  | データ型 | 例 | アイコン例 |
  |---|---|---|
  | 数値 | 出席番号（1〜50） | `pin` |
  | 単一の文字列 | 教員免許番号 | `text_fields` |
  | 複数の文字列 | 所属部活（最大3件） | `list` |
  | 日付 | 生年月日（未来日不可） | `calendar_month` |
  | コードマスターから選択 | 不登校歴＝コード「出欠状況」 | `data_object` |
- **バリデーション**：登録時に検証（型・正規表現 `^[A-Z]{2}-\d{8}$`・範囲・コード整合性）。
- **表示カテゴリ（タブ）**：項目を「基本情報／連絡先情報／学籍・部活／機微情報」等のタブに振り分ける。
  新カテゴリも作れる。
- **履歴保持**：オンにすると「いつからいつまでがその値か」を記録（適用期間付き）。
- **権限制限**：項目単位で閲覧/編集できる職種等を制限（後述）。
- **識別子設定**：CSVインポートで本人同定に使うキー（後述）。

参照実装：`attributes.html`（一覧＋定義モーダル＋識別子設定）、`codes.html`（コードマスター）。

## 7. 表示カテゴリ＝タブ

名簿の項目はカテゴリでグルーピングし、**タブで切り替えて表示**する。
- 例：基本情報（氏名・生年月日・性別・所属・クラス）／連絡先情報（住所・電話）／
  学籍・部活／機微情報（いじめ歴・不登校 等）。
- **機微情報タブ**はロックアイコン（`shield_lock`）＋監査記録の注記＋項目/選択肢ごとの権限制限。
- 参照実装：`roster.html`（一覧をカテゴリタブで切替）、`roster-detail.html`（個人詳細のタブ）。

## 8. 権限モデル（このシステム最大の強み）

詳細で柔軟な権限設定ができる。骨子は **サブジェクト × スコープ × オブジェクト × アクション** の条件式：

- **サブジェクト（誰が）**：職種／役割／**属性値の有無**（例：属性「部活顧問」に値がある）。AND条件可。
- **組織スコープ（どの範囲の）**：サブジェクトとオブジェクトの**関係**で動的に決まる
  （自分が担任のクラス／担当学校／同一市町村・同一学校種／配下の全組織／テナント全体）。
- **オブジェクト（何を）**：対象人物（生徒/教員/教委職員）の、どのカテゴリ/属性。
  さらに subject の属性で絞る（例：「サブジェクトの部活顧問と**同じ部**に所属する生徒」）。
- **アクション（どうできる）**：閲覧／編集／エクスポート／所属変更の承認 など。
- **割当機能**：そのポリシーがどの機能に効くか（名簿／フォーム対象者／データ利活用／CSV）。

代表ポリシー（`permissions.html` より）：
- 一般教員は「自分が担任のクラス」の生徒の基本情報を閲覧・編集できる。
- 属性「部活顧問」に値がある教員は、担当学校の同じ部の生徒の連絡先のみ閲覧できる。
- 校長は同一市町村・同一学校種の一般教員の連絡先を閲覧できる。
- 県教委の強い権限者は所属変更を承認フローなしで実行できる。

### 8.1 継承と上書き（下位優先）
- 全体設定は**テナント**で定義 → 各組織が**継承** → 組織が独自に**追加・上書き** → さらに配下へ波及。
- **下位の組織の設定が優先**（より下位で上書きされたらそれが勝つ）。
- 画面では：継承タグ `.inherit-tag`（テナント／組織独自／上書き）、上書き行は元ポリシーを
  取消線で示す、配下波及の警告callout。

### 8.2 属性・コード単位の権限
権限は属性項目・コード選択肢にも及ぶ：
- 属性「人物の特徴」は所属校の担任・学年主任のみ閲覧可。
- そのうち選択肢「いじめ加害者」（コード `P-04`）は**学年主任にしか表示されない**
  （条件に合わない人には選択肢自体が出ず、登録済みの値もマスクされる）。
- コードは `codes.html` で「表示制限（職種条件）」を選択肢ごとに設定。組織別の上書きも可。

## 9. CSVインポート（人に紐づく情報の主な入力経路）

UI編集も可能だが、基本は CSV 一括。流れ（`import.html` の4ステップが参照実装）：

1. **ファイル選択**：インポート種類（教職員/生徒/属性更新/所属変更/クラス編成）と対象スコープを選ぶ。
2. **項目マッピング**：CSVの各列 → 名簿のどの項目か を対応付け。
   役割タグ：識別子／一致確認／更新／スキップ。マッピングは保存・再利用できる。
3. **本人一致確認（同定）**：
   - あらかじめ **識別子**（職員番号・学籍番号・メール等）を設定しておく。
   - 識別子が一致した上で、**一致確認項目**（氏名＋生年月日 等）も照合し本人性を検証。
   - 結果は「一致（更新）／新規登録／**要確認**（識別子は合うが氏名等が相違）／**エラー**（識別子が名簿に無い）」。
4. **実行オプション**：実行タイミング（即時バックグラウンド／夜間バッチ／日時指定）、
   Workspace即時同期、ドライラン、そして **「所属変更の承認が不要」** チェック（後述）。
   → 「ジョブ登録 → ログで確認」へ。

CSVは即時完了でなく**バッチ/遅延でバックグラウンド処理**される。だから：
- ユーザーが自分の申請の反映状況を確認する **処理状況・ログ** ページが必要。
- エラーは**ダッシュボードで把握** → 必要なら**システム管理者へ問い合わせ**（`index.html` の問い合わせモーダル）。

## 10. 所属変更と承認フロー

人は所属を変えることがある。**自由ではなく原則承認制**。

```
① 申請（送出側の組織が作成）
② 承認（受入側の組織が承認）        ← 差戻しも可
③ 実行予約（指定日まで取消可能）
④ バッチ実行（指定日 04:00 に自動反映 → Workspace同期）
```

- 例：A校の先生が「7/1からXさんをB校へ」→ B校へ変更申請 → B校が承認 → 7/1にバッチで所属変更。
- 実行前なら取消可能。
- **例外**：権限の強い人（県教委等）は CSV から直接所属変更でき、**承認フロー不要**。
  CSVインポート時の「所属変更の承認が不要」チェックがこれにあたる（権限で許可された人のみ）。
- 参照実装：`transfers.html`（承認待ち/申請中/実行予約中/完了の4タブ＋申請・承認・取消モーダル）。

---

## 11. ジョブ／ログと問い合わせ

- CSVや年度更新などの重い処理は **ジョブ**（`#JOB-YYYYMMDD-NNN`）として登録・実行。
- 進捗・結果（成功/スキップ/エラー行）は **処理状況・ログ** で確認。
- エラーは**ダッシュボードの「要対応」**に出し、そこから **ログ確認** と **システム管理者へ問い合わせ**
  （エラーログ自動添付・個人情報マスク）ができる。

---

## 12. フォームサービス

教育委員会が対象者を絞ってアンケートを取る（参照実装：`form-builder.html`）。

- **対象者を権限条件式で限定**：種別＝教員 AND 組織スコープ＝テナント全体、など。
  対象者のみ閲覧・回答できる（配信時に権限エンジンで評価）。対象者数を即時表示。
- **集計**機能と、**結果の閲覧権限**もサブジェクト・オブジェクト関係で柔軟に
  （集計=匿名サマリーは教委職員、個票=同一校の管理職のみ、未回答者リスト=各校管理職、など）。
- **名簿への自動反映**：質問の回答を属性項目へ書き戻せる。
  例：「教員全員に自家用携帯番号を登録してほしい」→ メール（ログインアカウント）で本人同定し、
  回答値を属性「自家用携帯番号」へ反映（適用開始日＝回答日、反映前にバリデーション、結果はジョブ/ログ）。
