> このファイルは Claude Code / Cursor などの**エージェント型LLMが自動で読む**運用指示です。
> （`CLAUDE.md` という名前でも同じ。リポジトリ直下に置けばより広く自動読込されます。
> チャット型LLMには [`docs/04-chat-llm-prompt.md`](docs/04-chat-llm-prompt.md) を使ってください。）

このフォルダは教育委員会向けエンタープライズSaaS「**Composer EDU**」のHTML/JSモックです。
ページの**追加・拡張**を頼まれたら、既存の**世界観**（色・余白・密度・エンタープライズらしい落ち着き）と
**業務を深く考える姿勢**を引き継ぎつつ、**使う人にとって最も分かりやすいUI**を作ってください。

> 引き継ぐのは「世界観」と「思想」であって、**あなたの発想を縛る制約ではありません**。
> 既存コンポーネントは*出発点*。より分かりやすいUIのためなら、修正・拡張・新規作成して構いません。
> Fableの出力は優れた参照点ですが、絶対の正解ではない——**超えてよい**。
> ぶつかったときの優先順位は常に **人間にとっての分かりやすさ ＞ デザインの一貫性 ＞ 既存部品への忠実さ**。

## 要件プロンプト（`prompt/`）に基づく開発

`prompt/` には、モック構築の元となった要件が番号付きで格納されています（例：`001-初期構築.txt`）。
**どの番号まで適用済みかは [`prompt/applied-history.md`](prompt/applied-history.md) が唯一の正**である。


### プロンプトと成果物の関係

| 種類 | 役割 |
|---|---|
| **`prompt/NNN-*.txt`** | 番号付き要件本文。 |
| **`prompt/applied-history.md`** | 適用済み番号の履歴（日時・LLM・番号）。**着手前に必ず読む。** |
| **`docs/00`–`03`** | 適用済みプロンプトをもとに蒸留した運用ドキュメント。日常の拡張ではこちらが**実装の正**。 |


## 必ず守ること（思想と技術的な骨格）

外さないのはこの2つだけ。**残りは、より良いUIのためにあなたの裁量で動かしてよい。**

- **思想を再現する**【最重要】：要件はざっくり来る。質問で埋めず、ユーザーの背景・現場・権限環境を
  自分で深く推測し、必要な画面・状態・導線を一式設計して作り切る。「動く」でなく「分かる」を作る。
  docs/00 の署名的挙動①〜⑥（権限文章＋シミュレーター／マスク・自分の権限／非同期バッチ／履歴／
  一方向同期／リアルなダミーデータ）の**概念**を該当箇所で取り入れる（クラス名の再現ではなく*概念*の再現）。
- **技術的な骨格（“配管”）を壊さない**：これはデザインの制約ではなく、ページが動くための土台。
  各ページは `<main>` の中身だけ書く／`<body data-page="X">` を設定し `assets/app.js` を**末尾**で読み込む／
  ヘッダー・サイドバーのHTMLは手書きしない／新ページは `assets/app.js` の `NAV` 配列に項目を1つ追加
  （id を `data-page` と一致）。
  → これさえ守れば、**`<main>` の中のデザインは丸ごとあなたのもの**。

## 賢く使うこと（デザイン資産は“出発点”であって“天井”ではない）

- **まず既存資産を土台にする**——車輪の再発明を避け、ページ間の一貫性を保つため：
  - `assets/base.css` … デザイントークン＋コンポーネント（色・余白・部品の世界観のソース）。
  - `assets/app.js` … ヘッダー/サイドバーの自動注入＋`Composer.openModal/closeModal/toast`＋タブ等の配線。
- **ただし縛られない**：より分かりやすいUIのためなら、既存コンポーネントを**修正・拡張**してよいし、
  **新しいコンポーネントを作ってよい**。新しい色・パーツが要るなら既存トークンと調和させて足す。
  避けるべきは「既存と同じものを、ただ不揃いに作り直す」ことだけ。
- 判断に迷ったら **人間にとっての分かりやすさ ＞ デザインの一貫性 ＞ 既存部品への忠実さ**。

## 進め方（docs/03 準拠）
深掘り5問に自答 → （複数ページなら）動線を先に提案 → `_page-template.html` を複製して作成 →
`NAV` 追加 → docs/02 の部品を土台に構築（足りなければ拡張・新設してよい）→ 配線 →
docs/00 のセルフチェックで仕上げ。

## 完了時
- どんなユーザー・動線を想定し、権限/同期/履歴をどう見せたかを数行で要約。
- 追加した `NAV` 項目（id/icon/label/href）を明記。
- 可能なら新ページをブラウザで開き、サイドバー項目・タブ・モーダル・トーストの動作を確認。
